# 升豆游戏（ShengDouGame_Advance）Unity 项目架构文档

> 本文档面向新手开发者和有意参与此项目的游戏开发者。
> 项目基于 Unity 2022.3.62f2c1 (URP 14.0.12)，是 **MMORPG 手游** 框架。

---

## 一、概述

本项目是一个**完整的手游客户端框架**，使用 Unity URP 渲染管线，采用 **HybridCLR 热更新** + **Addressables 资源管理** + **FairyGUI UI 系统** 的技术组合。代码总量约 1245 个 C# 脚本。

### 核心目标
- **热更新能力**：HybridCLR 实现 C# 逻辑热更新
- **模块化架构**：Package 分包、模块可独立加载/卸载
- **MVC-like 分层**：Control（逻辑控制）- Model（数据）- View（UI 表现）
- **事件驱动**：全局事件系统解耦模块通信
- **资源热更新**：Addressables 资源远程下载 + 版本管理

### 4.2 核心设计模式：Control-Model-View 三层架构

这是项目**最核心的架构设计**。每个游戏功能（背包、任务、竞技场、邮件等）都遵守此模式：

#### BaseControl<T> — 逻辑控制器

- **单例模式**：`BaseControl<T>` 使用 `Lazy<T>` + 私有构造函数实现泛型单例
- **网络协议绑定**：`on<T1>(uint protoID, callback)` 注册网络回调，`SendNetMsg()` 发送网络消息
- **生命周期钩子**：
  - `onInit()` — 初始化
  - `onEventListener()` — 注册事件监听
  - `onLoginSuccess()` — 登录成功回调
  - `OnRefreshOnZero()` — 凌晨刷新
  - `onClear()` — 退出清理
- **跨程序集调用**：`InvokeAssemblyMethod()` / `OnAssembly()` — HybridCLR 热更程序集之间通信

#### BaseModel — 数据模型

- 负责数据的**存储、验证和变更通知**
- 通过 `Dispatch(EEventType)` 发送数据变更事件，通知 View 刷新

#### BaseView — UI 视图

- 继承自 FairyGUI 的 `Window` 类
- **属性声明式**：
  - `PackageName` / `ComponentName` — 绑定 FairyGUI 组件
  - `IsPreload` — 是否预加载
  - `EventList` — 事件监听列表
  - `mleftTabDatas` — 左侧 Tab 渲染数据
  - `mTopRenderData` — 顶部栏渲染数据
  - `IsFullScreen` / `ClickMaskHide` — UI 行为
- **生命周期**：`StartLoad` → `FinishLoad` → `OnCreate` → `Show` → `Hide` → `Destroy`
- **按钮分发**：自动遍历子按钮，统一回调 `OnButtonClick(GButton)`
- **红点系统**：`RegisterRedPoint()` 统一注册红点
- **UI 堆栈**：支持 `UIViewStack` 的 Push/Pop 导航（类 UINavigationController）

### 4.3 功能模块示例

以 **背包（Bag）** 为例的完整模块：

```
BagControl (BaseControl<BagControl>)
  ├── 监听网络协议 (bag相关 protoID)
  ├── 处理背包增删改逻辑
  └── 发送数据变更事件

BagModel (BaseModel)
  ├── 存储背包物品列表

  ├── 显示物品格子（GList）
  ├── 监听 OnBagUpdate 事件刷新界面

```

## 六、事件系统（EventSystem）

全局事件系统是整个项目的**通信总线**，解耦 Control、Model、View、Module。

### 架构

```
EEventType (enum)    →    EventManager (单例)
                              │
                    ┌─────────┼─────────┐
                    ↓         ↓         ↓
               ModuleManager  Controls  Views
                    ↓
              BaseModule 列表
```

### 特点

- **类型安全的事件参数**：支持 0~4 个泛型参数（`EventSysArgs<T1,T2,T3,T4>`）
- **对象池**：事件参数使用 `ClassPool` 回收复用，减少 GC
- **双广播路径**：`Dispatch()` 同时发送给 `ModuleManager.ExecuteNotificationHandle()` 和各注册监听器
- **自动清理**：监听数归零时，事件成员自动回收

### 常用事件（EEventType 示例）

| 事件 | 说明 |
|------|------|
| `LoginComplete` | 登录成功 |
| `EventRoleInfoUpdate` | 角色信息更新 |
| `OnRedPointUpdate` | 红点更新 |
| `Game_Update_Progress` | 游戏热更新进度 |
| `Mail_QueryMailInfoResp` | 邮件查询返回 |

## 七、网络通信层

### 双模式

1. **Socket 模式**（`NetworkManager`）：基于 TCP 的 Socket 通信
2. **WebSocket 模式**（`UnityWebSocketManager`）：WebSocket 通信（当前主要使用）

### 通信流程

```
客户端 → ProtoBuf 序列化 → 发送(protoID + bytes)
                                      ↓
服务端 → 处理 → ProtoBuf 响应
              ↓
客户端 UnityWebSocketManager 收到
              ↓
匹配 protoID → BaseControl.on<T>(protoID, callback)
 或
匹配 protoID → BaseModule.RegisterNetMsg(protoID)
              ↓
反序列化 → 回调处理 → Dispatch 事件通知 View
```

## 八、资源加载与热更新

### 8.1 Addressables 资源系统

- 所有可寻址资源通过 `Addressables` API 加载
- `FGUIAssetManager` 封装 FairyGUI 的资源加载
- `AssetNodeManager` 管理通用资源节点的引用计数和释放
- `RemoteAssetsUpdater` 实现远程资源更新

### 8.2 HybridCLR 热更新

- **AOT（主程序集）**：`AOTMainScript/` — 不可热更的基础代码
- **热更程序集**：各个 Package 编译为独立的 dll，通过 HybridCLR 加载
- 跨程序集通信：通过 `EventManager` + `Assembly_Invoke` 事件

### 8.3 预加载系统

`PreloadManager` + `PreloadOrder` 实现场景切换时的**资源预加载流程**：

```
场景切换时:
SceneManager.ChangeScene()
  → PreloadManager.ExecuteOrder(PreloadOrder)
     ├── 预加载 UI (FairyGUI 组件)
     ├── 加载场景 (Addressables.LoadSceneAsync)
     ├── 加载数据表
     └── 注册协议
```

## 十六、关键设计决策说明

### 为什么用 HybridCLR 而不是 Lua？
- 整个团队用 C#，避免 Lua 的桥接成本和性能损失
- HybridCLR 允许 C# 逻辑热更，开发效率高

### 为什么用 FairyGUI 而不是 uGUI？
- FairyGUI 有独立编辑器，美术和策划可直接编辑 UI
- 运行时性能优于 uGUI（尤其是列表和大规模 UI）

### 为什么用 Addressables？
- 支持远程资源下载和版本管理
- 依赖管理和引用计数自动化
- 与 AssetBundle 兼容性好
